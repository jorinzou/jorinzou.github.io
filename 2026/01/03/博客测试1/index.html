gdb命令总结：

info proc mappings # 查看进程内存映射
info args # 查看函数参数的地址
info locals # 查看局部变量的地址
info registers rsp rbp #查看栈地址范围
栈指针 ($rsp) 和基址指针 ($rbp)
info symbol 0x0000555555555378 #打印内存地址0x0000555555555378所在的内存段
x/5g 0x555555557d20 #连续打印内存值

1、成员变量内存布局

内存对齐：
内存对齐就是要求变量的起始地址，是4或者8的倍数上，具体是4还是8，决定于编译器的实现细节和硬件平台的要求

内存对齐是谁来处理？
编译器，也可以用#pragma pack或alignas关键字

为什么要内存对齐?
cpu访问内存地址时，对对齐的内存访问较快；如果不对齐，对讲较大的数据结构，可能存在跨缓存行或者内存页的存在，cpu加载时，就会多加载一些数据，导致加载变慢。
示例：若一个int（4字节）横跨两个64字节缓存行，需两次缓存加载，浪费带宽并增加延迟。

2、带函数成员 的对象内存布局

1、成员函数是否会占用内存大小？

不占用

2、成员函数存储的区域哪里，怎么看出来？

.text区域，通过gdb可看，info target
函数指针的地址是0x55555555485e，通过info target查看，
.text的地址范围是0x00005555555546a0 - 0x00005555555548f2 is .text

3、成员函数是怎么拿到成员变量并进行访问和赋值的？

通过gdb可以看出，成员函数，入参中，带着对象的this指针，this指针是整个对象的基地址，加上这些成员函数的偏移地址，计算出成员函数的base地址
然后就能访问到成员变量了。

3、私有成员的对象内存布局


私有成员和公有成员变量内存地址一致

私有成员的的访问属性在编译器有效，但可以在运行期，通过地址偏移，拿到私有成员变量的地址，进行赋值访问

4、静态成员变量内存布局

静态成员的生命周期，从被定义开始，直到程序结束

1、静态成员不占用类的内存空间

2、初始化的静态成员变量放在.data区，未初始化的放在.bss段

3、初始化的静态变量和未初始化的静态变量为什么要分开存储？
这里主要是考虑到二进制文件磁盘空间大小以及加载效率。在磁盘上，.data 占用实际的磁盘空间，因为它需要存储具体的初始值数据。.bss段不占用实际的存储空间，只需要在程序加载时由操作系统分配并清零相应的内存即可，这样可以减少可执行文件的大小。在程序启动时，操作系统可以快速地为.bss段分配内存并将其初始化为零，而无需从磁盘读取大量的零值数据，可以提高程序的加载速度。

4、静态成员方法
存储在.text区，类似于全局函数，只是是静态的，只能限定在类内使用
同时，静态成员函数不能直接访问类成员变量，从打印地址中，看到没有this指针

5、类继承的内存布局

继承类的内存布局中，派生类是包含了基类的成员变量，从gdb打印的地址中可以看出，是在同一块内存中。

派生类也包含了基类的成员方法，也是在同一块内存地址中。但成员方法和成员变量存储的不是同一个地址.
通过info target命令，.text段在内存中的位置范围
0x00005555555550a0 - 0x0000555555555315 is .text

基类函数地址
(void (*)(Basic * const, double)) 0x55555555525c Basic::setB(double)

派生类函数地址
(void (*)(Derived * const, int)) 0x55555555527e Derived::setC(int)

7、为什么多态要引用或者指针才能调用多态实现呢？

因为如果不是引用，赋值时，基类对象其实是对派生类一个拷贝构造，
这里的虚函数表指针，并没有指向派生类的虚函数地址，这会调用虚函数时，实际调用的是基类的虚函数。
